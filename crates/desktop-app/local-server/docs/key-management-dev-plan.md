# Key Management Phase 1 — 개발 계획서

> 설계 문서: [key-management-design.md](./key-management-design.md)
> GitHub Issue: [#45](https://github.com/tokamak-network/ethrex/issues/45)

## 목표

Manual Deploy (Path A)의 testnet 배포에서 **compose yaml에 프라이빗 키 값이 들어가지 않게** 한다.
compose 변수 치환 (`${VAR}`) + `.keys.env` 파일 분리 방식 사용.

**주소 파생 코드 제거** (L758-772):
```javascript
// 삭제: const deployerPk = deployerPrivateKey || "0x385c...";
// 삭제: const deployerAddress = new ethers.Wallet(deployerPk).address;
// 삭제: const committerPk = committerPkOpt || deployerPk;
// ... ethers.Wallet 관련 코드 모두 삭제
```

주소는 이제 파라미터로 직접 받으므로, 이 함수에서 ethers import 불필요.

**리턴값 변경:**
```javascript

// After — compose yaml만 리턴 (키 없음)
return yaml;
```

리턴 타입은 동일 (string). 호출부에서 키를 별도 처리.

### Task 2: `deployment-engine.js` — 주소 파생 + `.keys.env` 생성

#### 2-A. 헬퍼 함수 추가

파일 상단 (import 영역 근처)에 추가:

```javascript
const { ethers } = require("ethers");  // 이미 다른 곳에서 사용 중이면 확인

/**
 * 프라이빗 키에서 주소 파생
 */
function deriveAddresses({ deployerPk, committerPk, proofCoordinatorPk, bridgeOwnerPk }) {
  return {
    deployerAddress: new ethers.Wallet(deployerPk).address,
    committerAddress: new ethers.Wallet(committerPk).address,
    proofCoordinatorAddress: new ethers.Wallet(proofCoordinatorPk).address,
    bridgeOwnerAddress: new ethers.Wallet(bridgeOwnerPk).address,
  };
}

/**
 * .keys.env 파일 내용 생성
 */
function buildKeysEnvContent(keys, addresses) {
  return [
    "# Auto-generated by Tokamak Manager — DO NOT COMMIT",
    `ETHREX_DEPLOYER_L1_PRIVATE_KEY=${keys.deployerPk}`,
    `ETHREX_COMMITTER_L1_PRIVATE_KEY=${keys.committerPk}`,
    `ETHREX_PROOF_COORDINATOR_L1_PRIVATE_KEY=${keys.proofCoordinatorPk}`,
    `ETHREX_BRIDGE_OWNER_PK=${keys.bridgeOwnerPk}`,
    `ETHREX_BRIDGE_OWNER=${addresses.bridgeOwnerAddress}`,
    `ETHREX_ON_CHAIN_PROPOSER_OWNER=${addresses.bridgeOwnerAddress}`,
    `ETHREX_DEPLOYER_SEQUENCER_REGISTRY_OWNER=${addresses.deployerAddress}`,
    `ETHREX_DEPLOYER_COMMITTER_L1_ADDRESS=${addresses.committerAddress}`,
    `ETHREX_DEPLOYER_PROOF_SENDER_L1_ADDRESS=${addresses.proofCoordinatorAddress}`,
    "",
  ].join("\n");
}

/**
 * .keys.env 파일 쓰기 (chmod 600)
 */
function writeKeysEnvFile(deploymentId, keysEnvContent, customDir) {
  const deployDir = getDeploymentDir(deploymentId, customDir);
  const keysPath = path.join(deployDir, ".keys.env");
  fs.writeFileSync(keysPath, keysEnvContent, { mode: 0o600 });
  return keysPath;
}
```

> `getDeploymentDir`는 `compose-generator.js`에서 export된 함수. import 확인 필요.

### Task 3: `docker-local.js` — `--env-file` 지원

#### 3-A. `composeCmd()` 수정 (line 22)

**현재:**
```javascript
function composeCmd(projectName, composeFile, args) {
  return [DOCKER_BIN, "compose", "-p", projectName, "-f", composeFile, ...args];
}
```

**변경:**
```javascript
function composeCmd(projectName, composeFile, args, opts = {}) {
  const cmd = [DOCKER_BIN, "compose", "-p", projectName, "-f", composeFile];
  // .keys.env가 있으면 --env-file 추가
  if (opts.envFile) {
    cmd.push("--env-file", opts.envFile);
  }
  cmd.push(...args);
  return cmd;
}
```

#### 3-B. `runCompose()` 수정 (line 26)

**변경:**
```javascript
function runCompose(projectName, composeFile, args, opts = {}) {
  // compose 파일 옆에 .keys.env 파일이 있으면 자동 감지
  let envFile = opts.envFile;
  if (!envFile) {
    const keysPath = path.join(path.dirname(composeFile), ".keys.env");
    if (fs.existsSync(keysPath)) {
      envFile = keysPath;
    }
  }
  const [cmd, ...cmdArgs] = composeCmd(projectName, composeFile, args, { envFile });
  // ... 나머지 동일

```

> **자동 감지** 방식: `.keys.env`가 compose 파일과 같은 디렉토리에 있으면 자동으로 `--env-file`을 추가. 없으면 (Local 테스트 L1) 기존처럼 동작. 호출부 변경 최소화.

### Task 4: `docker-remote.js` — 원격 `--env-file` 지원

#### 4-A. `deployRemote()` 수정 (line 84)

**현재:**
```javascript
async function deployRemote(conn, projectName, composeContent, remoteDir) {
  await exec(conn, `mkdir -p ${remoteDir}`);
  await uploadFile(conn, composeContent, `${remoteDir}/docker-compose.yaml`);
  await exec(conn, `cd ${remoteDir} && docker compose -p ${projectName} pull`, { ... });
  await exec(conn, `cd ${remoteDir} && docker compose -p ${projectName} up -d`, { ... });
}
```

**변경:**
```javascript
async function deployRemote(conn, projectName, composeContent, remoteDir, opts = {}) {
  await exec(conn, `mkdir -p ${remoteDir}`);
  await uploadFile(conn, composeContent, `${remoteDir}/docker-compose.yaml`);

  // .keys.env 업로드 (있으면)
  if (opts.keysEnvContent) {
    await uploadFile(conn, opts.keysEnvContent, `${remoteDir}/.keys.env`);
    await exec(conn, `chmod 600 ${remoteDir}/.keys.env`);
  }

  const envFileFlag = opts.keysEnvContent ? `--env-file ${remoteDir}/.keys.env` : "";
  await exec(conn, `cd ${remoteDir} && docker compose ${envFileFlag} -p ${projectName} pull`, { ... });
  await exec(conn, `cd ${remoteDir} && docker compose ${envFileFlag} -p ${projectName} up -d`, { ... });
}
```

#### 4-B. 원격 start/stop 함수들

`startRemote()`, `stopRemote()`, `getStatusRemote()`, `getLogsRemote()` 등 원격 함수에서 `docker compose` 실행 시에도 `--env-file`이 필요한지 확인:

- `stopRemote()`: `docker compose stop` — env 불필요 → 변경 없음
- `startRemote()`: `docker compose up -d` — **env 필요** → `--env-file` 추가
- `destroyRemote()`: `docker compose down` — env 불필요 → 변경 없음
- `getStatusRemote()`: `docker compose ps` — env 불필요 → 변경 없음

`startRemote()`만 수정:
```javascript
async function startRemote(conn, projectName, remoteDir) {
  const keysEnvExists = await exec(conn, `test -f ${remoteDir}/.keys.env && echo yes || echo no`, { ignoreError: true });
  const envFileFlag = keysEnvExists.stdout.trim() === "yes" ? `--env-file ${remoteDir}/.keys.env` : "";
  await exec(conn, `cd ${remoteDir} && docker compose ${envFileFlag} -p ${projectName} up -d`, { timeout: 60000 });
}
```

## 주의 사항

1. **JavaScript 템플릿 리터럴 이스케이프**: compose yaml 안의 `${VAR}`는 Docker Compose 변수 치환이다. JS 템플릿 리터럴(`` ` ``)에서는 `\${VAR}`로 이스케이프해야 한다. 기존 코드에도 이 패턴이 있음 (L835: `\${ETHREX_BLOCK_PRODUCER_BLOCK_TIME:-5000}`).

2. **ethers.js import 위치 변경**: `compose-generator.js`에서 `ethers` import를 제거하고 `deployment-engine.js`로 이동. `deployment-engine.js`에 이미 ethers 의존성이 있는지 확인 필요.

3. **`getDeploymentDir` import**: `writeKeysEnvFile()`에서 사용하는 `getDeploymentDir`은 `compose-generator.js`에서 export되므로, `deployment-engine.js`에서 이미 import하고 있는지 확인.

4. **Local (테스트 L1) 하위 호환**: `generateComposeFile()`과 `generateRemoteComposeFile()`은 변경하지 않는다. 이 함수들의 하드코딩 키는 테스트용이므로 분리 불필요.

5. **원격 `docker compose up` 재시작 시 `--env-file`**: `provisionRemoteTestnet()`에서 직접 `remote.exec()`로 docker compose를 호출하는 부분이 있으면 모두 `--env-file` 추가 필요. `startRemote()` 외에 다른 호출 경로가 있는지 확인.
