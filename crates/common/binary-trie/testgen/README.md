# Binary Trie Test Vector Generator

This directory contains the Python script that generates the EIP-7864 reference
test vectors consumed by `tests/test_vectors.rs`.

## Requirements

- Python 3.10+
- `blake3` Python package

```bash
pip install blake3
```

## Regenerating vectors

The JSON files are git-ignored. Generate them locally with the make target from the repo root:

```bash
make binary-trie-vectors
```

Or invoke the script directly from the crate root or from this directory:

```bash
# From crate root (crates/common/binary-trie/):
python3 testgen/generate_vectors.py

# From the testgen/ directory:
python3 generate_vectors.py
```

Output files (all in `testgen/`):

- `test_vectors.json` — original 10 raw-insert vectors (Phase 1)
- `vectors_accounts.json` — account update sequences
- `vectors_storage.json` — storage slot sequences
- `vectors_codechunk.json` — code chunk sequences
- `vectors_negative.json` — absence-proof + selfdestruct-then-recreate cases

The generator is deterministic (PRNG seed `0xDEADBEEF42`); re-running produces byte-identical output.

## Running the vectors in Rust

The vector tests live in the workspace integration-test crate, not in this crate's `tests/`. Run via:

```bash
cargo test -p ethrex-test --test ethrex_tests trie::binary_vectors
```

The Rust harness reads each JSON at runtime via `std::fs::read_to_string` and skips with a `SKIP:` log line on missing files. So `cargo test` succeeds on a fresh checkout without Python installed; you only get the EIP-7864 cross-implementation oracle coverage after running `make binary-trie-vectors`.

## Vector format (`test_vectors.json`)

Each entry is:

```json
{
  "name": "descriptive name",
  "inserts": [
    { "key": "<64-hex-char 32-byte key>", "value": "<64-hex-char 32-byte value>" }
  ],
  "expected_root": "<64-hex-char 32-byte root>",
  "note": "(optional)"
}
```

The other JSON files (`vectors_*.json`) have their own schemas documented via the corresponding Rust deserialization structs in `tests/test_vectors.rs` (with `#[serde(deny_unknown_fields)]`).

Roots are computed by the Python EIP-7864 reference implementation using BLAKE3.
The Rust tests replay the ops and assert the roots match.

## Cross-validation

Because the test vectors are generated by the Python EIP-7864 reference and
verified by the Rust implementation, any divergence between them indicates a
bug in either the generator or the Rust trie code. The `expected_root` values
are ground truth; do not modify them without regenerating from a corrected
reference implementation.
