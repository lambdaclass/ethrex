# EIP-8250: Keyed Nonces for Frame Transactions

> **Spec target:** ethereum/EIPs `EIPS/eip-8250.md` (Draft, Standards Track: Core; requires EIP-8141). Read against `eips.ethereum.org` master on 2026-06-24. Pin the exact spec commit once it is frozen.

Implementation notes for EIP-8250 in ethrex. Builds on the EIP-8141 frame-transaction (type `0x06`) machinery and activates under the same `Fork::Hegota`.

## Overview

EIP-8250 replaces the single linear sender nonce of a frame transaction with a **keyed nonce** model: the envelope carries `nonce_keys` (1..=16 strictly-increasing `uint256` keys) plus a single `nonce_seq`. For every selected key, the protocol checks `current_nonce_seq(sender, key) == nonce_seq` and, on payment approval, advances each key's sequence by one. Key `0` is the legacy linear nonce domain (the account's `nonce`); non-zero keys live in the `NONCE_MANAGER` predeploy. This lets a shared sender run replay-independent transactions on disjoint key sets.

## Architecture

```
FrameTransaction envelope (crates/common/types/transaction.rs)
  nonce: u64                    ──►   nonce_keys: Vec<U256>  +  nonce_seq: u64
  (RLP, rkyv, sig_hash, serde, GenericTransaction, Transaction::nonce()->nonce_seq)

VM entry (crates/vm/levm/src/vm.rs)
  execute_frame_tx / run_frame_validation_prefix
    └─ for key in nonce_keys: current_nonce_seq(sender,key) == nonce_seq   (NonceMismatch otherwise)

APPROVE_PAYMENT / APPROVE_EXECUTION_AND_PAYMENT (opcode_handlers/frame_tx.rs)
    └─ consume_keyed_nonces(sender):
         key 0     → increment_account_nonce(sender)
         non-zero  → NONCE_MANAGER[slot(sender,key)] = nonce_seq+1  (+ KEYED_NONCE_FIRST_USE_GAS on 0→nonzero)

NONCE_MANAGER predeploy (crates/vm/system_contracts.rs, install in backends/levm/mod.rs)
    address 0x…8250, code 0x60006000fd (revert), nonce 1, installed at the Hegota boundary.
    slot(sender, key) = keccak256(left_pad_32(sender) || uint256_to_bytes32(key))
```

## TXPARAM index map (ethrex)

| Index | Value | Notes |
|-------|-------|-------|
| `0x01` | `nonce_seq` | re-pointed from EIP-8141's single `nonce` |
| `0x0B` | `len(signatures)` | **kept** (EIP-8141 allocation) |
| `0x0C` | pre-state legacy sender nonce | captured at tx entry (`FrameTxContext.legacy_sender_nonce`) |
| `0x0D` | `len(nonce_keys)` | |
| `0x0E` | `nonce_keys_hash` | `keccak256(be32(len) ‖ be32(k0) ‖ …)` |
| `0x10` | `nonce_keys[0]` | **relocated** (see divergence below) |

## Divergences from the draft spec (devnet)

1. **TXPARAM `nonce_keys[0]` at `0x10`, not `0x0B`.** The EIP-8250 draft assigns `0x0B` to `nonce_keys[0]`, but EIP-8141 already defines (and ethrex ships) `0x0B = len(signatures)` with no replacement index. ethrex keeps `0x0B = len(signatures)` and places `nonce_keys[0]` at the free index `0x10`. Raised upstream (an authoritative 8141-family TXPARAM registry is needed).
2. **`NONCE_MANAGER` address `0x…8250`.** The draft leaves the address `TBD`; ethrex uses the `0x…<EIP>` convention (matching `EXPIRY_VERIFIER = 0x8141`).
3. **Payment approval is forbidden inside an atomic batch.** EIP-8250 mandates that keyed-nonce consumption "MUST NOT be reverted by a later frame revert, by skipping frames, or by restoring an atomic-batch snapshot", and groups the five effects of a successful payment-scoped APPROVE (nonce consumption, payer recording, the balance debit, first-use gas, approval flags) as durable together. The "restore an atomic-batch snapshot" clause only bites when the payment APPROVE runs *inside* a batch (after the snapshot). Making all five effects survive that revert requires re-asserting the balance debit as an idempotent delta across arbitrarily nested reverts, coherent with the end-of-tx refund and EIP-7928 BAL — a mechanism the draft does not pin and that needs cross-client validation. Shipping only *part* of it (consumption + flags durable, debit reverted) would strand an authorization over an un-charged payer and mint the refund. So ethrex instead **rejects a payment-scoped APPROVE whose frame is in an atomic batch** (`apply_approve`, via `FrameTransaction::frame_is_in_atomic_batch`): the APPROVE reverts, `payer` stays unset, and the transaction is invalid. This matches the mempool, which already bans the batch flag in the validation prefix (where payment is granted); the consensus guard covers the crafted-block path the prefix rules do not reach. **The legitimate case is unaffected:** payment granted in a non-batch frame with a *later* SENDER atomic batch reverting keeps its consumption, because an independent frame's committed state is absorbed into the tx-level backup and is outside the batch's revert scope (covered by `eip8250_tests`). Full in-batch-payment durability is deferred to **Hegotá devnet + cross-client interop validation**.

## Validation rules

- **Nonce-key validity.** A `nonce_keys` list that contains key `0` is rejected unless it is exactly `[0]` — key 0 maps to the account's linear nonce, so it may only appear as the sole key. This closes a double-consumption hole where one transaction could consume the account nonce via both key 0 and a keyed slot. Enforced in `validate_static_constraints` (with a unit test); composes with the existing 1..=16, strictly-increasing, `nonce_seq != u64::MAX` checks.
- **NONCE_MANAGER activation.** At the Hegota boundary the predeploy at `0x…8250` is installed following the spec's 3-case rule: nonce = `max(existing_nonce, PREDEPLOY_NONCE)`, balance preserved, idempotent on re-run (early-returns once the expected runtime bytecode is present, so exactly one account update is produced at the first Hegota block). Case 3 (an address that already carries code/storage) is spec-undefined and is not special-cased.

## Mempool policy

The public mempool admits only **key-0** frame transactions (`nonce_keys == [0]`). The spec explicitly permits this minimal policy; multi-key / non-zero-key admission needs per-keyset pending/replacement tracking that is deferred. Non-zero-key frame transactions can still be included by a block builder and are validated in full at block execution.
