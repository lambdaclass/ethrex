# EIP-8141: Frame Transactions

> **Spec target:** ethereum/EIPs `EIPS/eip-8141.md` @ commit `fe0940cae2457d8a190471cc57812af96698c125` (pinned 2026-06-03).
> All implementation comments referencing other spec commits are historical; this pin governs.

Implementation notes for EIP-8141 in ethrex. Covers the architecture, where the spec required interpretation, and known limitations.

## Overview

Frame transactions (type `0x06`) replace the ECDSA signature of traditional transactions with a sequence of **frames** — independent execution steps that collectively authenticate and pay for the transaction. Authentication happens on-chain via the `APPROVE` opcode rather than off-chain via secp256k1.

This enables:

- **Abstract authentication**: Any on-chain verification scheme (WebAuthn/P256, multisig, social recovery) can authorize transactions.
- **Separated sender and payer**: One account proves identity, a different account pays for gas.
- **Batched execution**: Multiple operations in a single atomic transaction.

## Architecture

```
 eth_sendRawTransaction(0x06 || rlp(frame_tx))
          │
          ▼
┌─────────────────────┐
│   RPC layer         │  Decode 0x06, create SendRawTransactionRequest::Frame
│   rpc/types/tx.rs   │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│   Mempool           │  Validate: nonce, fees, frame count, gas limit,
│   blockchain.rs     │  signatures. Skip balance/EOA (payer unknown until
│                     │  execution). Reject blob-carrying frame txs (no
│                     │  sidecar transport yet).
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│   Payload builder   │  Include in block; remove from mempool on failure
│   payload.rs        │  (frame tx failures are deterministic)
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│   VM entry point    │  Detect TxType::Frame → execute_frame_tx()
│   vm.rs             │  Bypasses standard prepare_execution/finalize_execution
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│   Frame loop        │  For each frame:
│   vm.rs             │    1. Set caller per mode (ENTRY_POINT or sender)
│                     │    2. Create CallFrame with frame's gas_limit
│                     │    3. Execute via run_execution() (or default code)
│                     │    4. Commit or revert state per frame result
│                     │    5. A reverted VERIFY frame invalidates the tx;
│                     │       unroll atomic batches on failure
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│   Post-execution    │  Verify payer set, compute gas refund,
│   vm.rs             │  pay coinbase, build ExecutionReport
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│   Receipt           │  Consensus payload: [cumulative_gas_used, payer,
│   receipt.rs        │  [frame_receipt...]] with frame_receipt =
│                     │  [status, gas_used, logs]
└─────────────────────┘
```

## Transaction Format

### Wire encoding

```
0x06 || rlp([chain_id, nonce, sender, frames, signatures,
             max_priority_fee_per_gas, max_fee_per_gas,
             max_fee_per_blob_gas, blob_versioned_hashes])
```

The `signatures` list is part of the wire/RLP body (it is committed by the
transaction and is the same field that `compute_sig_hash` operates over). Each
frame is RLP-encoded as a six-element tuple:

```
[mode, flags, target, gas_limit, value, data]
```

- `mode` — 0 = DEFAULT, 1 = VERIFY, 2 = SENDER (3-255 reserved).
- `flags` — bits 0-1 are the APPROVE scope restriction, bit 2 is the atomic-batch
  flag, bits 3-7 reserved (must be zero).
- `target` — a 20-byte address or RLP null (empty bytes) for "use tx.sender".
- `gas_limit` — per-frame gas allocation.
- `value` — wei transferred from `sender`; must be zero unless the frame is SENDER.
- `data` — calldata for the frame's top-level call.

Each signature is RLP-encoded as:

```
[scheme, signer, msg, signature]
```

Where `scheme` is 0 (SECP256K1, `signature = v||r||s`, 65 bytes) or 1 (P256,
`signature = r||s||qx||qy`, 128 bytes), `signer` is a 20-byte address, and `msg`
is either empty (the signature is over the canonical sig-hash) or an explicit
non-zero 32-byte digest. Raw `signature` bytes are intentionally not
EVM-introspectable.

**Source**: `crates/common/types/transaction.rs` — `FrameTransaction`, `Frame`,
`FrameSignature`, their `RLPEncode`/`RLPDecode` implementations (golden RLP
vectors cross-check the byte layout).

### Key differences from standard transactions

| Field | Standard TX | Frame TX |
|-------|-------------|----------|
| Authentication | ECDSA signature (v, r, s) | Outer `signatures` list + on-chain APPROVE opcode |
| Sender | Recovered from signature | Explicit `sender` field |
| Value | Transfer amount | Per-frame `value` (only SENDER frames may be non-zero) |
| Data | Single calldata blob | Per-frame data arrays |
| Gas limit | Single value | Intrinsic + per-frame cost + calldata cost + sig-verify cost + Σ(frame.gas_limit) |
| Nonce increment | Before execution | During APPROVE payment (scope 0x1 or 0x3) |
| Balance deduction | Before execution | During APPROVE payment (scope 0x1 or 0x3) |

### Signature hash (sig_hash)

The sig_hash is the canonical digest a signature with empty `msg` is signed over.
It commits to the entire transaction — including all frame data — with one
exception: the raw `signature` bytes of any signature whose `msg` is empty are
elided (replaced with empty bytes), because a signature over `compute_sig_hash`
cannot commit to its own bytes. Signatures with an explicit 32-byte `msg` keep
their bytes (they sign that digest, not this hash).

```
sig_hash = keccak256(0x06 || rlp([chain_id, nonce, sender, frames,
                                  elided_signatures, ...]))
```

Where `elided_signatures` is the `signatures` list with each empty-`msg`
signature's raw `signature` field zeroed to empty bytes; all frame data
(including VERIFY frames and expiry deadlines) is committed verbatim. Validation
data added after the sender signs should be carried as elided signature bytes,
not as mutable frame data.

**Source**: `FrameTransaction::compute_sig_hash()` in `transaction.rs`.

## Frame Modes

### DEFAULT (mode = 0)

Regular execution. The caller (msg.sender) is `ENTRY_POINT` (`0x00..00aa`). Not static. Used for operations that don't need sender authority — deploying contracts via CREATE2, calling permissionless functions.

### VERIFY (mode = 1)

Validation execution. The caller is `ENTRY_POINT`. Executed in **static mode** — no state modifications allowed except via APPROVE. A VERIFY frame that **reverts** invalidates the entire transaction (whether or not it is part of an atomic batch). A VERIFY frame that **succeeds without calling APPROVE is valid** — the expiry verifier frame is exactly this case. APPROVE is how a VERIFY frame grants sender and/or payer approval, but it is not a requirement for the frame itself to be considered valid.

### SENDER (mode = 2)

Authorized execution. The caller is `tx.sender` — the contract sees the call as coming from the account itself. Not static. May carry a non-zero `value`. Requires that sender has already been approved (APPROVE with scope `APPROVE_EXECUTION` (0x2) or `APPROVE_EXECUTION_AND_PAYMENT` (0x3) called in a prior frame). Used for the actual operations: token transfers, contract calls, etc.

**Source**: `vm.rs` frame loop for caller/static assignment and VERIFY enforcement.

## New Opcodes

The implementation adds six opcodes: APPROVE (0xAA), TXPARAM (0xB0),
FRAMEDATALOAD (0xB1), FRAMEDATACOPY (0xB2), FRAMEPARAM (0xB3), and SIGPARAM
(0xB4). Outside a frame transaction, all six halt exceptionally (the
`frame_tx_context` is `None`).

**Source for opcode bytes**: `crates/vm/levm/src/opcodes.rs`.

### APPROVE (0xAA)

**Stack**: `[offset, length, scope]` (offset on top)

Exits the current frame's execution context successfully (like RETURN) and
updates transaction-scoped approval state based on the `scope` operand. `scope`
is a bitmask:

| Scope | Name | Effect |
|-------|------|--------|
| 0x0 | `APPROVE_NONE` | No approval allowed — APPROVE with this scope is rejected. When a frame's flags restrict the allowed scope to 0 (`APPROVE_SCOPE_NONE`), the frame may not approve anything; default code reverts/halts. |
| 0x1 | `APPROVE_PAYMENT` | Increments `tx.sender`'s nonce, deducts the tx cost from the executing contract's balance, and records it as `payer`. |
| 0x2 | `APPROVE_EXECUTION` | Sets `sender_approved = true`. Requires `resolved_target == tx.sender`. |
| 0x3 | `APPROVE_EXECUTION_AND_PAYMENT` | Both payment and execution approval in one atomic step. Requires `resolved_target == tx.sender`. |

The deducted payment cost is `total_gas_limit × effective_gas_price + blob_fees`
(see [Gas Accounting](#gas-accounting)). `APPROVE_SCOPE_MASK` is an alias for
`0x3`.

The data at `memory[offset..offset+length]` becomes the frame's output (like
RETURN).

**Gas cost**: Memory expansion only (same as RETURN — no base opcode cost).

**Invariants enforced**:
- `scope` must be a non-zero value `≤ 3` and a subset of the frame's
  `flags`-derived allowed scope (bits 0-1). A zero or `> 3` scope, or a scope
  not permitted by the flags, is an exceptional halt.
- Cannot re-approve a scope already set (double payment via existing `payer`, or
  double execution via existing `sender_approved` → exceptional halt).
- Scopes 0x2 and 0x3 require `resolved_target == tx.sender` (a mismatch reverts
  the frame).
- The executing contract (call frame's `to` address) must equal the frame's
  resolved target.
- Payment scopes (0x1, 0x3) revert the frame if the payer balance underflows.

**Source**: `crates/vm/levm/src/opcode_handlers/frame_tx.rs` (`OpApproveHandler`,
`apply_approve`).

### TXPARAM (0xB0)

**Stack input**: `[param_id]`
**Stack output**: `[value]`
**Gas cost**: 2

Pops a single `param_id` and pushes a 32-byte word for the requested
transaction-scoped parameter:

| param_id | Returns |
|----------|---------|
| 0x00 | `0x06` (transaction type) |
| 0x01 | nonce |
| 0x02 | sender (right-aligned in 32 bytes) |
| 0x03 | max_priority_fee_per_gas |
| 0x04 | max_fee_per_gas |
| 0x05 | max_fee_per_blob_gas |
| 0x06 | max cost (max_fee_per_gas × total_gas_limit + blob cost — the **maximum**, not the effective amount charged) |
| 0x07 | len(blob_versioned_hashes) |
| 0x08 | sig_hash |
| 0x09 | len(frames) |
| 0x0A | currently executing frame index |
| 0x0B | len(signatures) |

Any other `param_id` is an exceptional halt.

**Source**: `frame_tx.rs` (`OpTxParamHandler`, `load_tx_param`).

### FRAMEDATALOAD (0xB1)

**Stack input**: `[offset, frameIndex]` (offset on top)
**Stack output**: `[word]`
**Gas cost**: 3 (matches CALLDATALOAD)

Loads one 32-byte word from `frames[frameIndex].data` starting at byte `offset`,
zero-padding past the end (CALLDATALOAD semantics). An out-of-bounds
`frameIndex` is an exceptional halt.

**Source**: `frame_tx.rs` (`OpFrameDataLoadHandler`).

### FRAMEDATACOPY (0xB2)

**Stack input**: `[memOffset, dataOffset, length, frameIndex]` (memOffset on top)
**Gas cost**: 3 + 3 × ceil(length / 32) + memory expansion (CALLDATACOPY model)

Copies `length` bytes from `frames[frameIndex].data` (starting at `dataOffset`)
into memory at `memOffset`, zero-padding past the end. An out-of-bounds
`frameIndex` is an exceptional halt (checked before the zero-length early
return).

**Source**: `frame_tx.rs` (`OpFrameDataCopyHandler`).

### FRAMEPARAM (0xB3)

**Stack input**: `[param, frameIndex]` (frameIndex on top)
**Stack output**: `[value]`
**Gas cost**: 2

Returns a 32-byte word for the requested frame-scoped parameter of
`frames[frameIndex]`:

| param | Returns |
|-------|---------|
| 0x00 | resolved target (frame.target, or tx.sender if null) |
| 0x01 | gas_limit |
| 0x02 | mode |
| 0x03 | flags |
| 0x04 | len(data) |
| 0x05 | status (0 = failure, 1 = success, 3 = skipped) — exceptional halt for the current or a future frame |
| 0x06 | allowed_scope (`flags & 0x03`) |
| 0x07 | atomic_batch flag (`(flags >> 2) & 1`, returns 0 or 1) |
| 0x08 | value |

Any other `param`, or an out-of-bounds `frameIndex`, is an exceptional halt.

**Source**: `frame_tx.rs` (`OpFrameParamHandler`).

### SIGPARAM (0xB4)

**Stack input**: `[param, signatureIndex]` (signatureIndex on top)
**Stack output**: `[value]`
**Gas cost**: 2

Returns signature-scoped metadata for `signatures[signatureIndex]`. The raw
`signature` bytes are intentionally not exposed.

| param | Returns |
|-------|---------|
| 0x00 | effective signer address |
| 0x01 | scheme |
| 0x02 | msg (0 when empty — the canonical sig-hash case — else the 32-byte digest) |
| 0x03 | len(signature) |

Any other `param`, or an out-of-bounds `signatureIndex`, is an exceptional halt.

**Source**: `frame_tx.rs` (`OpSigParamHandler`).

## Gas Accounting

### Total gas limit

`FrameTransaction::total_gas_limit()` computes:

```
total_gas_limit =
      FRAME_TX_INTRINSIC_COST                       # 15000
    + len(frames) * FRAME_TX_PER_FRAME_COST         # 475 each
    + calldata_cost(rlp(frames)) + calldata_cost(rlp(signatures))
    + signature_verification_cost                   # 2800 per SECP256K1, 6700 per P256
    + Σ(frame.gas_limit)
```

The calldata cost charges 4 gas per zero byte and 16 gas per non-zero byte over
the RLP-encoded `frames` and `signatures` (matching standard transaction
calldata costing). `FRAME_TX_PER_FRAME_COST = 475` covers a fixed CALL context
overhead (100) plus `G_log` (375) for each frame's receipt sub-entry.

**Source**: `FrameTransaction::total_gas_limit()` and
`signature_verification_cost()` in `transaction.rs`.

### Fee charged to the payer

When APPROVE grants payment (scope 0x1 or 0x3), the payer is charged:

```
tx_fee = total_gas_limit × effective_gas_price + blob_fees
```

at the **effective** gas price (not the max). `blob_fees = len(blob_versioned_hashes)
× 131072 × blob_base_fee`. Unused gas is refunded to the payer at the same
effective rate at the end of the transaction, so no gas value is silently
destroyed; blob fees are non-refundable (EIP-4844 burn semantics).

`TXPARAM(0x06)` reports the **maximum** cost (`max_fee_per_gas`-based, including
blob cost), which is a separate quantity from the effective fee actually charged.

**Source**: `apply_approve` / `compute_tx_cost` in `frame_tx.rs`.

### During execution

Each frame runs with its own gas_limit. Gas consumed is tracked per-frame. A frame
that reverts via the `REVERT` opcode is charged its actual `gas_used` (standard EVM
semantics); a frame that fails with an exceptional halt (e.g. out-of-gas, invalid
opcode) is charged its full gas_limit. Frames skipped by a failed atomic batch are
not executed and their gas is refunded. (The atomic-batch-*revert* path currently
charges full gas_limit uniformly, which is inconsistent with this single-frame rule
and is spec-silent — see Spec Gaps §11.)

The intrinsic gas (everything in `total_gas_limit` other than `Σ(frame.gas_limit)`)
is always consumed — it is not part of any frame's gas budget and is
non-refundable.

### Refund

```
frame_gas_used    = total_gas_used - intrinsic_gas
gas_refund        = Σ(frame.gas_limit) - frame_gas_used
refund_amount     = gas_refund × effective_gas_price
```

The refund goes to the payer (the address recorded by APPROVE with payment scope
0x1 or 0x3).

**Source**: `vm.rs` post-execution refund/coinbase block.

### Coinbase payment

```
priority_fee   = effective_gas_price - base_fee_per_gas
coinbase_fee   = priority_fee × total_gas_used
```

**Source**: `vm.rs` post-execution refund/coinbase block.

## Execution Flow

### 1. Validation

- Frame count: 1 ≤ len(frames) ≤ `FRAME_TX_MAX_FRAMES` (64)
- Static constraints: reserved modes/flags, per-frame value rule, signature
  schemes/`msg` lengths, expiry-verifier frame shape
- Nonce: must match sender's account nonce
- Fees: max_priority_fee ≤ max_fee, max_fee ≥ base_fee
- All outer `signatures` validate before any frame executes

Notable omission vs standard transactions: **no balance check, no EOA check, no nonce increment**. These are handled by APPROVE during execution instead of during validation.

### 2. Context initialization

Creates `FrameTxContext` with:
- `sender_approved: false`, `payer_address: None`
- Computes `sig_hash` (empty-`msg` signature bytes elided; frame data committed)
- Calculates intrinsic gas, starts `total_gas_used` at intrinsic cost

### 3. Frame loop

For each frame in order:

1. **Set context**: Update `current_frame_index`, reset `approve_called_in_current_frame`.
2. **Resolve target**: `frame.target.unwrap_or(tx.sender)`.
3. **Set caller and static mode** per frame mode (see [Frame Modes](#frame-modes)).
4. **Create CallFrame**: Fresh stack, memory, and the frame's gas_limit.
5. **Push substate backup**: State isolation per frame (and a batch-level backup at atomic-batch entry).
6. **Execute**: `run_execution()`, or run the built-in default code when the resolved target has no code.
7. **Handle result**:
   - Success → commit substate, collect logs
   - Failure → revert substate, charge full gas_limit; if inside an atomic batch, unroll the batch (roll back to its entry snapshot and mark remaining batch frames SKIPPED with status 0x3)
8. **Store result**: `(status, gas_used, logs)` in `frame_results`.
9. **VERIFY enforcement**: A reverted VERIFY frame sets `tx_invalid = true` and breaks. A VERIFY frame that succeeds without APPROVE is valid.
10. **Clear transient storage** between frames.

### 4. Post-execution

- Check `payer_address.is_some()` (otherwise TX invalid)
- Compute and apply gas refund to payer at the effective gas price
- Pay coinbase (`priority_fee × total_gas_used`)
- Derive top-level `succeeded` from SENDER frame statuses
- Build `ExecutionReport` with frame_results and payer_address

### 5. Invalid transactions

If any of these fail, the transaction is rejected entirely (no state changes):
- A VERIFY frame reverts (batched or not)
- No payer set after all frames execute
- A SENDER frame executes before sender approval

Invalid frame transactions that reach the payload builder are **removed from the mempool** since their failure is deterministic.

## Receipt Format

Frame receipts use distinct encodings for consensus and for internal storage.

**Consensus payload** (receipts-trie root, and served over eth/68) —
`encode_inner_with_bloom`:

```
0x06 || rlp([cumulative_gas_used, payer, [frame_receipt, ...]])
frame_receipt = [status, gas_used, logs]
```

There is no top-level `succeeded`, no bloom, and no aggregated `logs`. On decode,
`succeeded` is **derived**: true iff every frame's status is `0x1` (success).
`status` codes: `0x0` failure, `0x1` success, `0x3` skipped (frame skipped by a
failed atomic batch). The `payer` is the address recorded by APPROVE payment
(zero address ⇒ `None`).

**eth/69** serves receipts through the `RLPEncode for Receipt` impl
(`encode_inner`), not `encode_inner_with_bloom`. This carries the same fields but
puts the type as the first element *inside* the RLP list rather than as a raw
EIP-2718 prefix byte — `rlp([0x06, cumulative_gas_used, payer, [frame_receipt, ...]])`
— consistent with how eth/69 encodes every typed receipt. The two wire layouts
are therefore NOT byte-identical for frame receipts; only eth/68 matches the
receipts-trie bytes.

**Internal storage encoding** — `encode_storage` — additionally persists the
top-level `succeeded` flag and the aggregated `logs`, which the consensus layout
omits but which `eth_getLogs` / `eth_getTransactionReceipt` need:

```
0x06 || rlp([succeeded, cumulative_gas_used, logs, payer, [frame_receipt, ...]])
```

**RPC response** (`eth_getTransactionReceipt`) includes `payer` and
`frameReceipts` as additional JSON fields when the transaction is type 0x06.

**Source**: `crates/common/types/receipt.rs` (`FrameReceipt`,
`encode_inner_with_bloom`, `encode_storage`, `encode_inner`),
`crates/networking/rpc/types/receipt.rs`.

## Mempool Handling

Frame transactions receive special treatment in mempool validation (`blockchain.rs`):

1. **Balance check skipped**: The payer is unknown until execution (it's determined by which address calls APPROVE with a payment scope). We can't verify balance at mempool time.
2. **EOA check skipped**: For the same reason — a non-existent sender is valid if a payer will fund the execution. Nonce sanity is still applied to such senders against their implied nonce of 0 (see point 7), rather than skipped.
3. **Static + signature validation up front**: `validate_static_constraints` runs at admission, the outer `signatures` are verified, and the combined frame `data` size is bounded.
4. **`MAX_VERIFY_GAS` signature cap**: the signature-verification cost (spec rule #6, `MAX_VERIFY_GAS = 100_000`) is checked before any per-signature crypto runs; a tx whose signatures alone exceed the budget is rejected. This is a sound subset of the full validation-prefix budget check (see Known Limitations).
5. **Hegota fork gating**: frame txs are rejected before the Hegota fork activates.
6. **Expiry drop**: a frame tx whose expiry-verifier deadline has passed is dropped.
7. **Blob frame txs rejected**: a frame tx carrying `blob_versioned_hashes` is rejected at admission — there is no sidecar transport for frame-tx blobs yet (consensus still accounts for blob cost if another builder includes one).
8. **Standard checks still apply**: Nonce ordering (including for not-yet-existent senders, whose implied nonce is 0), fee validation, gas limit vs block limit, intrinsic gas coverage, chain ID.

### P2P propagation

Frame transactions **are** propagated over P2P, but **announce-only**: like
EIP-4844 blob txs, they are advertised by hash via `NewPooledTransactionHashes`
(never full-body broadcast in a `Transactions` message), and the full body is
served on request through `GetPooledTransactions` →
`PooledTransactions` as `P2PTransaction::FrameTransaction`.

**Source**: `crates/networking/p2p/tx_broadcaster.rs`,
`crates/networking/p2p/rlpx/eth/transactions.rs`.

## Spec Gaps and Design Decisions

The EIP-8141 specification leaves several behaviors underspecified. This section documents how ethrex fills those gaps.

### 1. ORIGIN opcode behavior

**Spec gap**: The spec doesn't define what the `ORIGIN` opcode (0x32) should return during frame execution. In standard transactions, ORIGIN always returns the EOA that signed the transaction.

**ethrex decision**: `env.origin` is set to the frame's **caller** — `ENTRY_POINT (0xaa)` for DEFAULT/VERIFY frames, `tx.sender` for SENDER frames (`vm.rs`).

**Rationale**: Frame transactions don't have a single signing EOA. Setting ORIGIN to the frame's caller is consistent with the frame execution model. Setting it to tx.sender for all frames would leak sender identity to DEFAULT/VERIFY frame targets, which may not be desirable.

**Risk**: Contracts that use `tx.origin == msg.sender` as an EOA check will behave differently in frame transactions. This is unlikely to cause issues in practice since `tx.origin` for authorization is already a known anti-pattern.

### 2. APPROVE gas cost

**Spec gap**: The spec says APPROVE "exits context" but doesn't specify its gas cost.

**ethrex decision**: APPROVE uses the same gas model as RETURN — memory expansion cost only, no base opcode cost (`frame_tx.rs`, `gas_cost::exit_opcode`).

**Rationale**: APPROVE is functionally equivalent to RETURN (copies memory to output, halts execution) with the additional side effect of updating approval state. The approval state update is a transaction-level bookkeeping operation, not an EVM storage operation, so no additional gas is charged.

### 3. Transient storage between frames

**Spec gap**: The spec doesn't state whether transient storage (EIP-1153 TSTORE/TLOAD) persists across frames.

**ethrex decision**: Transient storage is **cleared between frames** (`vm.rs`).

**Rationale**: Frames are independent execution contexts. Allowing transient storage to leak between frames would create an implicit communication channel that could interact with frame ordering in surprising ways. Clearing it is the conservative choice.

### 4. APPROVE output data accessibility

**Spec gap**: APPROVE copies `memory[offset..length]` to output, but the spec doesn't define how subsequent frames can access this data.

**ethrex decision**: The output data is stored in the CallFrame's output buffer during execution, but `frame_results` only records `(status, gas_used, logs)` — the output bytes are **discarded** (`vm.rs`).

**Impact**: `FRAMEPARAM` param 0x05 returns a past frame's status, but there's no way to read a VERIFY frame's output data from a later frame. If a use case requires passing structured data from a VERIFY frame to a SENDER frame, it would need to use events (logs) or a separate storage-based communication pattern.

**Possible future work**: Add output bytes to `frame_results` and expose via a new `FRAMEPARAM` param.

### 5. Access list pre-warming

**Spec gap**: Standard transactions pre-warm the sender and `to` address per EIP-2929. The spec doesn't define pre-warming behavior for frame targets.

**ethrex decision**: No pre-warming is performed for frame targets. The `execute_frame_tx()` path bypasses `prepare_execution()` which handles access list initialization (`vm.rs`).

**Impact**: The first SLOAD/SSTORE on each frame target pays the cold access surcharge (2100 gas). This makes frame transactions slightly more expensive than they would be if targets were pre-warmed. The sender address is also cold.

**Possible future work**: Pre-warm `tx.sender` and all unique `frame.target` addresses before the frame loop begins.

### 6. Substate gas refunds (SSTORE refunds) within frames

**Spec gap**: The spec defines gas refunds for unused frame gas but doesn't address EVM-level gas refunds (e.g., SSTORE clearing refund from EIP-3529).

**ethrex decision**: The `substate.refunded_gas` accumulates normally within each frame, but the final refund calculation (`vm.rs`) only considers unused frame gas limits — it does **not** add SSTORE refunds on top.

**Impact**: SSTORE clearing refunds that would normally reduce gas_used in standard transactions are effectively lost in frame transactions. Gas accounting is simpler but slightly less favorable to users who clear storage.

### 7. Receipt encoding in receipt trie (resolved)

**Spec gap**: The spec defines a receipt format `[cumulative_gas_used, payer, [frame_receipt, ...]]` but the earlier ethrex implementation did not commit the frame-specific fields into the receipts-trie encoding.

**ethrex decision (current)**: For type 0x06, `encode_inner_with_bloom` now emits the EIP-8141 consensus payload `0x06 || rlp([cumulative_gas_used, payer, [frame_receipt, ...]])` — there is no top-level `succeeded`, no bloom, and no aggregated `logs`; `succeeded` is derived from the frame statuses on decode. eth/68 serves these exact bytes; eth/69 serves the same fields via `encode_inner` (type as the first list element, not a prefix byte), so the two wire layouts are not byte-identical for frame receipts (see [Receipt Format](#receipt-format)). The receipts-trie root commits to the payer and every per-frame receipt. The internal storage encoding (`encode_storage`) keeps `succeeded` and aggregated `logs` for RPC/`eth_getLogs`. See [Receipt Format](#receipt-format).

### 8. ENTRY_POINT address collision

**Spec gap**: The spec defines `ENTRY_POINT = 0x00..00aa` as the caller for DEFAULT and VERIFY frames. It does not reserve this address or prevent deployment at it.

**ethrex decision**: Uses `Address::from_low_u64_be(0xaa)` as-is (`vm.rs`).

**Impact**: If a contract exists at address `0xaa` on the chain, DEFAULT/VERIFY frames would appear to originate from that contract. On the demo chain this is not an issue. On mainnet, this address should be either reserved or protected from deployment.

### 9. Failed frame behavior and top-level `succeeded`

**Spec gap**: The spec says "if frame reverts, discard state changes and continue." Two things are underspecified: (a) whether execution continues to subsequent frames, and (b) what the top-level receipt `status` should be.

**ethrex decision**: A reverted non-VERIFY frame discards its state changes but execution continues to the next frame; the frame is recorded as failed (`0x0`) in `frame_results`. A reverted VERIFY frame instead invalidates the whole transaction (it is authentication, so the tx is excluded). The transaction is still included and gas is paid when a SENDER/DEFAULT frame reverts, matching the standard Ethereum model where `status: 0` means execution reverted.

The top-level `succeeded` is **derived identically on encode and decode**: `true` iff *every* frame's status is `0x1` (success), i.e. `false` if any frame (of any mode) reverted or was skipped (`vm.rs` sets the VM result this way; `receipt.rs` re-derives the same on decode). This unification is mandatory, not a stylistic choice: the consensus `frame_receipt` (`[status, gas_used, logs]`) carries **no frame mode**, so a node decoding receipts from the wire or the receipts trie cannot tell SENDER frames apart from DEFAULT/VERIFY ones. An all-frames rule is therefore the only definition the executing node and a wire/trie-decoding node can compute identically; a SENDER-only rule would make a freshly-executed receipt's `status` disagree with the same receipt decoded from consensus bytes (note: `succeeded` is itself not in the consensus encoding, only the per-frame statuses are, so this only affects the derived JSON-RPC `status`, never the receipts root).

**Rationale**: In standard Ethereum transactions, `status: 0` means execution failed — it does not mean the transaction wasn't included. The same applies to frame transactions. Per-frame receipts in `frame_receipts` provide granular detail for batch operations where some frames may succeed and others fail; the aggregate `succeeded` simply reports whether the whole transaction executed cleanly.

### 10. Double-call protection for APPROVE

**Spec gap**: The spec implicitly assumes APPROVE is called once per approval type, but doesn't explicitly state what happens if a later frame re-approves a scope that is already set.

**ethrex decision**: Re-approving an already-set scope is an exceptional halt — `APPROVE_PAYMENT` (0x1) fails if `payer` is already set, `APPROVE_EXECUTION` (0x2) fails if `sender_approved` is already true, and `APPROVE_EXECUTION_AND_PAYMENT` (0x3) fails if either is already set (`frame_tx.rs`, `apply_approve`).

**Rationale**: Re-approving is likely a bug in the frame transaction construction. Failing loudly prevents subtle issues like a second payer accidentally overriding the first.

### 11. Atomic-batch revert: gas and status of rolled-back frames

**Spec gap**: When a frame in an atomic batch fails, the spec rolls back the batch's state to the point before the batch began and skips the remaining frames (refunding their gas). It is silent on: (a) the gas charged to frames that already executed-and-succeeded *earlier* in the same batch; (b) the gas charged to the failing frame when it failed via `REVERT` (vs an exceptional halt); and (c) the receipt `status` of rolled-back-but-succeeded frames.

**ethrex behavior (current)**: on batch revert, every frame in the batch — the earlier succeeded ones and the failing one — is recorded as `0x0` (FAILURE) and charged its full `gas_limit`. This **diverges from the single-frame path**, which charges a `REVERT`-failing frame its actual `gas_used` (only an exceptional halt is charged the full `gas_limit`); the same `REVERT`ing frame is therefore billed differently inside vs outside a batch.

**Status — UNRESOLVED, pending cross-client confirmation.** Because the spec is silent, the canonical per-frame **gas** (actual `gas_used` vs full `gas_limit`) and **status** (`0x0` FAILURE vs `0x3` SKIPPED) for rolled-back batch frames are a consensus decision that must be aligned with the reference / other clients before being pinned. Both feed `total_gas_used` → the block header `gasUsed`, the receipts trie (per-frame status + gas), and the payer refund, so the choice is consensus-visible. Recommended target once confirmed: charge actual `gas_used` for a `REVERT` (full `gas_limit` only for an exceptional halt), matching the single-frame path, and pick one status for rolled-back succeeded frames. Not changed unilaterally here to avoid introducing a *different* divergence.

### 12. Frame `gas_limit` bound is `2**63-1`, not `2**64-1` (intentional stricter validity rule)

**Spec rule**: EIP-8141 static validation allows `frame.gas_limit <= 2**64-1` and cumulative frame gas `<= 2**64-1`.

**ethrex behavior**: `validate_static_constraints` (`transaction.rs`) rejects any `frame.gas_limit > i64::MAX` (`2**63-1`) and any cumulative `> i64::MAX`. This is **stricter than the spec** — a frame tx the spec considers statically valid (a `gas_limit` in `[2**63, 2**64-1]`) is rejected. The cap exists because the EIP-8037 state-gas dimension is tracked as `i64`; keeping the static bound at `i64::MAX` guarantees the downstream gas accounting can never overflow.

**Status — documented divergence, effectively unreachable.** Any `gas_limit >= 2**63` exceeds every real block gas limit by orders of magnitude and is rejected by the gas-limit-vs-block-limit check regardless, so no spec-conformant builder could produce a block that exercises the difference. Kept as a deliberate stricter rule rather than relaxing to `2**64-1` (which would require making the entire gas path `u64`-clean) until there is a concrete need.

## Known Limitations

### In-memory frame transaction context

The `FrameTxContext` is stored on the VM struct and accessed via `Option<FrameTxContext>`. This means:

- Frame transaction opcodes (APPROVE, TXPARAM, FRAMEDATALOAD, FRAMEDATACOPY, FRAMEPARAM, SIGPARAM) halt exceptionally when executed outside a frame transaction — they check for `frame_tx_context.is_some()`.
- The context is created fresh for each frame transaction and dropped after execution.
- No frame transaction state persists across transactions.

### Blob frame transactions are rejected at the mempool

A frame tx carrying `blob_versioned_hashes` is rejected at mempool admission because there is no sidecar transport for frame-tx blobs yet, so this node will not admit or build one.

**Consensus accounting is asymmetric and the mempool gate is load-bearing.** Block *import* fully accounts for frame blobs: `verify_blob_gas_usage` counts their `GAS_PER_BLOB * len` toward `blob_gas_used` and the per-block blob cap, and `execute_frame_tx` enforces the EIP-4844 `max_fee_per_blob_gas >= base_blob_fee` validity rule. But the block *build* path (`payload.rs` `apply_transaction`) does **not** yet add frame-tx blob gas to `header.blob_gas_used` — only EIP-4844 txs are routed through `apply_blob_transaction`. So a block this node built containing a blob-carrying frame tx would undercount `header.blob_gas_used` and then fail its own `verify_blob_gas_usage`. This is currently **unreachable** because the mempool gate above rejects every blob-carrying frame tx and the builder sources only from the mempool. If that gate is ever lifted (sidecar transport added), the build path must route frame blobs through blob accounting first.

### Mempool validation-prefix simulation (implemented)

The full ERC-7562-style validation-prefix simulation is implemented as a local peer policy. Admission runs the following checks in order, all in addition to the static and signature checks above:

1. **Prefix recognition and structural validation**: the frame list is walked to identify the recognized shapes (`self_verify`, `deploy+self_verify`, `only_verify+pay`, `deploy+only_verify+pay`), and structural rules (frame modes, scope restrictions, gas budget) are validated. Unrecognized or invalid prefixes are rejected at admission.
2. **Validation-trace simulation**: a dedicated `ValidationObserver` on the VM enforces ERC-7562-style banned-opcode and storage-dependency rules during a throwaway prefix-only execution against the canonical head state. Banned opcodes (TIMESTAMP outside the expiry verifier, BALANCE, SELFBALANCE, BLOCKHASH, ORIGIN, COINBASE, GASPRICE, NUMBER, PREVRANDAO, GASLIMIT, BASEFEE, BLOBHASH, BLOBBASEFEE, SELFDESTRUCT, INVALID, TLOAD, TSTORE; SSTORE/CREATE/CREATE2 only inside a deploy frame; GAS only immediately before a call instruction) cause the tx to be rejected with `FrameTxValidationFailed`.
3. **Paymaster availability accounting**: the simulation identifies the payer (paymaster) and reserves the tx's `max_cost` against the paymaster's on-chain balance. Concurrent pending reservations are summed so multiple sponsored txs cannot collectively overdraw the paymaster.
4. **Post-block revalidation**: after each new canonical block, pending frame txs are re-simulated against the updated head state and evicted if they no longer pass. Expiry-based eviction (deadline now behind the new block's timestamp) runs first without a full re-simulation.

**This is a local peer policy**: it never affects consensus or block building. It may over-reject (conservative) and must never under-reject.

**Known interim limitations (OQ resolutions)**:

- **OQ1 (canonical paymaster bytecode)**: no canonical paymaster bytecode hash is pinned in the EIP draft or any reference implementation. All paymasters are therefore treated as non-canonical, making the effective limit `MAX_PENDING_TXS_USING_NON_CANONICAL_PAYMASTER = 1` pending sponsored tx per paymaster. A clearly-marked hook (`FRAME_CANONICAL_PAYMASTER_CODE_HASH` const + `is_canonical_paymaster` fn in `mempool.rs`) is in place to flip this once the canonical bytecode is resolved upstream.
- **OQ3 (pending withdrawal amount)**: ethrex has no paymaster withdrawal queue. The pending withdrawal amount is treated as 0 in the availability formula: `available = balance - reserved_cost`. This is conservative (the formula under-computes available funds only if the paymaster has a real pending withdrawal, which cannot occur here).
- **Reorg re-add of reservations**: when a block is re-orged out, frame txs from the re-orged block are not automatically re-admitted with their reservations restored. A subsequent re-submission re-runs full admission and re-reserves. This is out of scope (ties to existing TODO #797).

**Policy scope**: this is a local peer policy. It runs only during mempool admission and post-block revalidation. It never influences block building, execution, or consensus. A frame tx admitted by this policy and included in a block by a different builder is still executed correctly.

### No access list support

Frame transactions don't have an access list field. Standard EIP-2930 access lists allow pre-declaring storage slots to reduce gas costs. Adding access list support to frame transactions would require extending the RLP format.

### Calldata gas is computed from RLP-encoded frames

The calldata gas component of intrinsic gas (`total_gas_limit()` in `transaction.rs`) re-encodes the frames and signatures to RLP to count bytes. This is functionally correct but allocates a temporary buffer on every call. For gas limit validation in hot paths, this could be cached.
