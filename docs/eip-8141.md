# EIP-8141: Frame Transactions

Implementation notes for EIP-8141 in ethrex. Covers the architecture, where the spec required interpretation, and known limitations.

## Overview

Frame transactions (type `0x06`) replace the ECDSA signature of traditional transactions with a sequence of **frames** — independent execution steps that collectively authenticate and pay for the transaction. Authentication happens on-chain via the `APPROVE` opcode rather than off-chain via secp256k1.

This enables:

- **Abstract authentication**: Any on-chain verification scheme (WebAuthn/P256, multisig, social recovery) can authorize transactions.
- **Separated sender and payer**: One account proves identity, a different account pays for gas.
- **Batched execution**: Multiple operations in a single atomic transaction.

## Architecture

```
 eth_sendRawTransaction(0x06 || rlp(frame_tx))
          │
          ▼
┌─────────────────────┐
│   RPC layer         │  Decode 0x06, create SendRawTransactionRequest::Frame
│   rpc/types/tx.rs   │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│   Mempool           │  Validate: nonce, fees, frame count, gas limit
│   blockchain.rs     │  Skip balance/EOA check (payer unknown until execution)
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│   Payload builder   │  Include in block; remove from mempool on failure
│   payload.rs        │  (frame tx failures are deterministic)
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│   VM entry point    │  Detect TxType::Frame → execute_frame_tx()
│   vm.rs:556-560     │  Bypasses standard prepare_execution/finalize_execution
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│   Frame loop        │  For each frame:
│   vm.rs:668-787     │    1. Set caller per mode (ENTRY_POINT or sender)
│                     │    2. Create CallFrame with frame's gas_limit
│                     │    3. Execute via run_execution()

│                     │    5. Enforce VERIFY → APPROVE invariant

          │
          ▼
┌─────────────────────┐
│   Receipt           │  Receipt includes payer address and
│   receipt.rs        │  per-frame receipts [status, gas_used, logs]
└─────────────────────┘
```

## Transaction Format

### Wire encoding

```
0x06 || rlp([chain_id, nonce, sender, frames, max_priority_fee_per_gas,
             max_fee_per_gas, max_fee_per_blob_gas, blob_versioned_hashes])
```

Each frame is RLP-encoded as:

```
[mode, target, gas_limit, data]
```

Where `target` is a 20-byte address or RLP null (empty bytes) for "use tx.sender".

**Source**: `crates/common/types/transaction.rs` — `FrameTransaction`, `Frame`, their `RLPEncode`/`RLPDecode` implementations.

### Key differences from standard transactions

| Field | Standard TX | Frame TX |
|-------|-------------|----------|
| Authentication | ECDSA signature (v, r, s) | On-chain APPROVE opcode |
| Sender | Recovered from signature | Explicit `sender` field |
| Value | Transfer amount | Always zero (transfers happen via frame execution) |
| Data | Single calldata blob | Per-frame data arrays |
| Gas limit | Single value | Intrinsic + calldata cost + Σ(frame.gas_limit) |
| Nonce increment | Before execution | During APPROVE (scope 1 or 2) |
| Balance deduction | Before execution | During APPROVE (scope 1 or 2) |

### Signature hash (sig_hash)

The sig_hash is what gets signed by the authenticating account. It commits to the entire transaction except the VERIFY frame data (since that data contains the signature itself):

```
sig_hash = keccak256(0x06 || rlp([chain_id, nonce, sender, elided_frames, ...]))
```

Where `elided_frames` replaces each VERIFY frame's data with empty bytes while preserving all other frame data unchanged.

**Source**: `FrameTransaction::compute_sig_hash()` in `transaction.rs:1797-1827`.

## Frame Modes

### DEFAULT (mode = 0)

Regular execution. The caller (msg.sender) is `ENTRY_POINT` (`0x00..00aa`). Not static. Used for operations that don't need sender authority — deploying contracts via CREATE2, calling permissionless functions.

### VERIFY (mode = 1)

Validation execution. The caller is `ENTRY_POINT`. Executed in **static mode** — no state modifications allowed except via APPROVE. The frame's data field is elided from the sig_hash, allowing it to contain the signature without creating a circular dependency. Must call APPROVE before returning; if it doesn't, the entire transaction is invalid.

### SENDER (mode = 2)

Authorized execution. The caller is `tx.sender` — the contract sees the call as coming from the account itself. Not static. Requires that sender has already been approved (APPROVE scope 0 or 2 called in a prior VERIFY frame). Used for the actual operations: token transfers, contract calls, etc.

**Source**: `vm.rs:680-695` for caller/static assignment, `vm.rs:779-783` for VERIFY enforcement.

## New Opcodes

### APPROVE (0xAA)

**Stack**: `[offset, length, scope]` (offset on top)

Exits the current frame's execution context and updates transaction-level approval state.

| Scope | Effect |
|-------|--------|
| 0x0 | **Sender approval.** Sets `sender_approved = true`. The executing contract must be `tx.sender`. |
| 0x1 | **Payer approval.** Increments `tx.sender`'s nonce. Deducts `max_tx_cost` from the executing contract's balance. Sets `payer_approved = true`. Requires sender already approved. |
| 0x2 | **Combined.** Both sender and payer approval in one call. Executing contract must be `tx.sender`. |

`max_tx_cost = max_fee_per_gas × total_gas_limit + len(blob_hashes) × 131072 × max_fee_per_blob_gas`

The data at `memory[offset..offset+length]` becomes the frame's output (like RETURN).

**Gas cost**: Memory expansion only (same as RETURN — no base opcode cost).

**Invariants enforced**:
- Cannot call APPROVE twice for the same scope (double sender_approved or double payer_approved → error).
- Scope 0 and 2 require `frame_target == tx.sender`.
- Scope 1 requires `sender_approved == true`.
- The executing contract (call frame's `to` address) must be the frame's target.

**Source**: `crates/vm/levm/src/opcode_handlers/frame_tx.rs:32-158`.

### TXPARAMLOAD (0xB0)

**Stack input**: `[param_id, index]` (param_id on top)
**Stack output**: `[value]`
**Gas cost**: 3

Returns a 32-byte word for the requested transaction parameter:

| param_id | index | Returns |
|----------|-------|---------|
| 0x00 | 0 | `0x06` (transaction type) |
| 0x01 | 0 | nonce |
| 0x02 | 0 | sender (right-aligned in 32 bytes) |
| 0x03 | 0 | max_priority_fee_per_gas |
| 0x04 | 0 | max_fee_per_gas |
| 0x05 | 0 | max_fee_per_blob_gas |
| 0x06 | 0 | max_cost (gas cost + blob cost) |
| 0x07 | 0 | len(blob_versioned_hashes) |
| 0x08 | 0 | sig_hash |
| 0x09 | 0 | len(frames) |
| 0x10 | 0 | current_frame_index |
| 0x11 | frame_idx | frames[idx].target (or tx.sender if null) |
| 0x12 | frame_idx | frames[idx].data (first 32 bytes; 0 for VERIFY) |
| 0x13 | frame_idx | frames[idx].gas_limit |
| 0x14 | frame_idx | frames[idx].mode |
| 0x15 | frame_idx | frames[idx].status (only for past frames) |

Invalid param_id or out-of-bounds index → exceptional halt.
Param 0x15 for current or future frames → exceptional halt.

**Source**: `frame_tx.rs:160-343`.

### TXPARAMSIZE (0xB1)

**Stack input**: `[param_id, index]`
**Stack output**: `[size]`
**Gas cost**: 3

Returns the byte size of a parameter. All scalar parameters return 32. Param 0x12 (frame data) returns the actual data length (0 for VERIFY frames).

**Source**: `frame_tx.rs:182-366`.

### TXPARAMCOPY (0xB2)

**Stack input**: `[param_id, index, dest_offset, src_offset, length]` (param_id on top)
**Gas cost**: 3 + 3 × ceil(length / 32) + memory expansion

Copies transaction parameter data into EVM memory. Zero-pads if `src_offset + length` exceeds the data. Follows the same copy gas model as CALLDATACOPY.

**Source**: `frame_tx.rs:204-249`.

## Gas Accounting

### Intrinsic gas

```
total_gas_limit = 15000 + calldata_cost(rlp_encode(frames)) + Σ(frame.gas_limit)
```

The calldata cost charges 4 gas per zero byte and 16 gas per non-zero byte of the RLP-encoded frames — matching standard transaction calldata costing.

**Source**: `FrameTransaction::total_gas_limit()` in `transaction.rs:1830-1841`.

### During execution

Each frame runs with its own gas_limit. Gas consumed is tracked per-frame. If a frame reverts or halts with error, it is charged its full gas_limit.

The intrinsic gas (15000 + calldata cost) is always consumed — it is not part of any frame's gas budget and is non-refundable.

### Refund

```
frame_gas_used    = total_gas_used - intrinsic_gas
gas_refund        = Σ(frame.gas_limit) - frame_gas_used
refund_amount     = gas_refund × effective_gas_price
```

The refund goes to the payer (the address that called APPROVE with scope 1 or 2).

**Source**: `vm.rs:816-825`.

### Coinbase payment

```
priority_fee   = effective_gas_price - base_fee_per_gas
coinbase_fee   = priority_fee × total_gas_used
```

**Source**: `vm.rs:828-832`.

## Execution Flow

### 1. Validation (vm.rs:597-642)

- Frame count: 1 ≤ len(frames) ≤ 1000
- Nonce: must match sender's account nonce
- Fees: max_priority_fee ≤ max_fee, max_fee ≥ base_fee

Notable omission vs standard transactions: **no balance check, no EOA check, no nonce increment, no value transfer**. These are handled by APPROVE during execution instead of during validation.

### 2. Context initialization (vm.rs:644-665)

Creates `FrameTxContext` with:
- `sender_approved: false`, `payer_approved: false`
- Computes `sig_hash` (VERIFY data elided)
- Calculates intrinsic gas, starts `total_gas_used` at intrinsic cost

### 3. Frame loop (vm.rs:668-787)

For each frame in order:

1. **Set context**: Update `current_frame_index`, reset `approve_called_in_current_frame`.
2. **Resolve target**: `frame.target.unwrap_or(tx.sender)`.
3. **Set caller and static mode** per frame mode (see [Frame Modes](#frame-modes)).
4. **Create CallFrame**: Fresh stack, memory, and the frame's gas_limit.
5. **Push substate backup**: State isolation per frame.
6. **Execute**: `run_execution()` — the standard LEVM opcode dispatch loop.
7. **Handle result**:
   - Success → commit substate, collect logs
   - Failure → revert substate, charge full gas_limit
8. **Store result**: `(success, gas_used, logs)` in `frame_results`.
9. **VERIFY enforcement**: If VERIFY frame didn't call APPROVE, set `tx_invalid = true` and break.
10. **Clear transient storage** between frames.

### 4. Post-execution (vm.rs:789-845)

- Check `payer_approved == true` (otherwise TX invalid)
- Compute and apply gas refund to payer
- Pay coinbase
- Build `ExecutionReport` with frame_results and payer_address

### 5. Invalid transactions

If any of these fail, the transaction is rejected entirely (no state changes):
- VERIFY frame doesn't call APPROVE
- No payer approved after all frames execute
- SENDER frame executes before sender_approved

Invalid frame transactions that reach the payload builder are **removed from the mempool** since their failure is deterministic (the VERIFY frame data is bound to the specific transaction).

## Receipt Format

Frame transaction receipts extend the standard receipt with two additional fields:

```
Standard: [tx_type, succeeded, cumulative_gas_used, logs]
Frame:    [tx_type, succeeded, cumulative_gas_used, logs, payer, [frame_receipts...]]
```

Each frame receipt: `[status, gas_used, logs]`

The `payer` field is the address that called APPROVE with scope 1 or 2.

**RPC response** (`eth_getTransactionReceipt`) includes `payer` and `frameReceipts` as additional JSON fields when the transaction is type 0x06.

**Source**: `crates/common/types/receipt.rs`, `crates/networking/rpc/types/receipt.rs`.

## Mempool Handling

Frame transactions receive special treatment in mempool validation (`blockchain.rs:2534-2610`):

1. **Balance check skipped**: The payer is unknown until execution (it's determined by which address calls APPROVE scope 1/2). We can't verify balance at mempool time.
2. **EOA check skipped**: For the same reason — a non-existent sender is valid if a payer will fund the execution.
3. **Standard checks still apply**: Nonce ordering, fee validation, gas limit vs block limit, intrinsic gas coverage, chain ID.

Frame transactions are **not gossiped via P2P** (`blockchain.rs:2678-2682`). They are only accepted via `eth_sendRawTransaction` RPC.

## Spec Gaps and Design Decisions

The EIP-8141 specification leaves several behaviors underspecified. This section documents how ethrex fills those gaps.

### 1. ORIGIN opcode behavior

**Spec gap**: The spec doesn't define what the `ORIGIN` opcode (0x32) should return during frame execution. In standard transactions, ORIGIN always returns the EOA that signed the transaction.

**ethrex decision**: `env.origin` is set to the frame's **caller** — `ENTRY_POINT (0xaa)` for DEFAULT/VERIFY frames, `tx.sender` for SENDER frames (`vm.rs:698`).

**Rationale**: Frame transactions don't have a single signing EOA. Setting ORIGIN to the frame's caller is consistent with the frame execution model. Setting it to tx.sender for all frames would leak sender identity to DEFAULT/VERIFY frame targets, which may not be desirable.

**Risk**: Contracts that use `tx.origin == msg.sender` as an EOA check will behave differently in frame transactions. This is unlikely to cause issues in practice since `tx.origin` for authorization is already a known anti-pattern.

### 2. APPROVE gas cost

**Spec gap**: The spec says APPROVE "exits context" but doesn't specify its gas cost.

**ethrex decision**: APPROVE uses the same gas model as RETURN — memory expansion cost only, no base opcode cost (`frame_tx.rs:57-61`).

**Rationale**: APPROVE is functionally equivalent to RETURN (copies memory to output, halts execution) with the additional side effect of updating approval state. The approval state update is a transaction-level bookkeeping operation, not an EVM storage operation, so no additional gas is charged.

### 3. Transient storage between frames

**Spec gap**: The spec doesn't state whether transient storage (EIP-1153 TSTORE/TLOAD) persists across frames.

**ethrex decision**: Transient storage is **cleared between frames** (`vm.rs:786`).

**Rationale**: Frames are independent execution contexts. Allowing transient storage to leak between frames would create an implicit communication channel that could interact with frame ordering in surprising ways. Clearing it is the conservative choice.

### 4. APPROVE output data accessibility

**Spec gap**: APPROVE copies `memory[offset..length]` to output, but the spec doesn't define how subsequent frames can access this data.

**ethrex decision**: The output data is stored in the CallFrame's output buffer during execution, but `frame_results` only records `(success, gas_used, logs)` — the output bytes are **discarded** (`vm.rs:776-777`).

**Impact**: TXPARAMLOAD param 0x15 returns frame success/failure status, but there's no way to read a VERIFY frame's output data from a later frame. If a use case requires passing structured data from a VERIFY frame to a SENDER frame, it would need to use events (logs) or a separate storage-based communication pattern.

**Possible future work**: Add output bytes to `frame_results` and expose via a new TXPARAMLOAD param_id or through TXPARAMCOPY.

### 5. Access list pre-warming

**Spec gap**: Standard transactions pre-warm the sender and `to` address per EIP-2929. The spec doesn't define pre-warming behavior for frame targets.

**ethrex decision**: No pre-warming is performed for frame targets. The `execute_frame_tx()` path bypasses `prepare_execution()` which handles access list initialization (`vm.rs:597`).

**Impact**: The first SLOAD/SSTORE on each frame target pays the cold access surcharge (2100 gas). This makes frame transactions slightly more expensive than they would be if targets were pre-warmed. The sender address is also cold.

**Possible future work**: Pre-warm `tx.sender` and all unique `frame.target` addresses before the frame loop begins.

### 6. Substate gas refunds (SSTORE refunds) within frames

**Spec gap**: The spec defines gas refunds for unused frame gas but doesn't address EVM-level gas refunds (e.g., SSTORE clearing refund from EIP-3529).

**ethrex decision**: The `substate.refunded_gas` accumulates normally within each frame, but the final refund calculation (`vm.rs:818-820`) only considers unused frame gas limits — it does **not** add SSTORE refunds on top.

**Impact**: SSTORE clearing refunds that would normally reduce gas_used in standard transactions are effectively lost in frame transactions. Gas accounting is simpler but slightly less favorable to users who clear storage.

### 7. Receipt encoding in receipt trie

**Spec gap**: The spec defines a receipt format `[cumulative_gas_used, payer, [frame_receipts]]` but doesn't specify whether the bloom-based encoding (used for receipt trie roots) should include frame-specific fields.

**ethrex decision**: The bloom-based receipt encoding (`encode_inner_with_bloom` in `receipt.rs:102-117`) uses the standard format `[succeeded, cumulative_gas_used, bloom, logs]` and does **not** include `payer` or `frame_receipts`.

**Impact**: The receipt trie root does not commit to per-frame receipt data or the payer address. This data is available via RPC but is not part of the consensus-critical receipt encoding. A client syncing from receipt proofs would not be able to verify frame-level data.

**Note**: The non-bloom encoding (`encode_inner` in `receipt.rs:80-100`) does include `payer` and `frame_receipts` for type 0x06, so these fields survive storage and RPC serialization.

### 8. ENTRY_POINT address collision

**Spec gap**: The spec defines `ENTRY_POINT = 0x00..00aa` as the caller for DEFAULT and VERIFY frames. It does not reserve this address or prevent deployment at it.

**ethrex decision**: Uses `Address::from_low_u64_be(0xaa)` as-is (`vm.rs:659`).

**Impact**: If a contract exists at address `0xaa` on the chain, DEFAULT/VERIFY frames would appear to originate from that contract. On the demo chain this is not an issue. On mainnet, this address should be either reserved or protected from deployment.

### 9. Failed SENDER frame behavior

**Spec gap**: The spec says "if frame reverts, discard state changes and continue." It's not entirely clear whether a failed SENDER frame should stop the transaction or continue to subsequent frames.

**ethrex decision**: A reverted SENDER frame discards its state changes but execution continues to the next frame. The frame is recorded as failed in `frame_results`. The overall transaction still succeeds as long as VERIFY and payer approval invariants are met (`vm.rs:732-756`).

**Rationale**: This matches the spec's "discard and continue" language and allows batch operations where some individual operations may fail without invalidating the entire transaction.

### 10. Double-call protection for APPROVE

**Spec gap**: The spec implicitly assumes APPROVE is called once per scope type, but doesn't explicitly state what happens if a second VERIFY frame calls APPROVE scope 0 after it's already been set.

**ethrex decision**: Calling APPROVE for a scope that's already been set is an error (`frame_tx.rs:71-73` for scope 0, `83-85` for scope 1, `112-113` for scope 2).

**Rationale**: Re-approving is likely a bug in the frame transaction construction. Failing loudly prevents subtle issues like a second payer accidentally overriding the first.

## Known Limitations

### In-memory frame transaction context

The `FrameTxContext` is stored on the VM struct and accessed via `Option<FrameTxContext>`. This means:

- Frame transaction opcodes (APPROVE, TXPARAMLOAD, etc.) fail with `InvalidOpcode` when executed outside a frame transaction — they check for `frame_tx_context.is_some()`.
- The context is created fresh for each frame transaction and dropped after execution.
- No frame transaction state persists across transactions.

### No P2P propagation

Frame transactions are not included in `P2PTransaction` and cannot be gossiped. They must be submitted directly to a node via RPC. This is appropriate for the demo but would need to be addressed for production use with distributed block building.

### No access list support

Frame transactions don't have an access list field. Standard EIP-2930 access lists allow pre-declaring storage slots to reduce gas costs. Adding access list support to frame transactions would require extending the RLP format.

### Calldata gas is computed from RLP-encoded frames

The calldata gas component of intrinsic gas (`total_gas_limit()` in `transaction.rs:1831-1837`) re-encodes the frames to RLP to count bytes. This is functionally correct but allocates a temporary buffer on every call. For gas limit validation in hot paths, this could be cached.
