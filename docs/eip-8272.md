# EIP-8272: Recent Roots for Frame Transactions

> **Spec target:** ethereum/EIPs `EIPS/eip-8272.md` (Draft, Standards Track: Core; requires EIP-7843, EIP-8141). Read against `eips.ethereum.org` master on 2026-06-24. Pin the exact spec commit once it is frozen.

Implementation notes for EIP-8272 in ethrex. Builds on EIP-8141 frame transactions and EIP-7843 `SLOTNUM` (both already present); activates under `Fork::Hegota`.

## Overview

EIP-8272 lets a frame transaction declare verified **recent roots** — `(source_id, slot, root)` tuples — in its signed envelope, so validation code (e.g. privacy proofs) can reference a recent commitment without reading another account's mutable storage. References are checked for validity at block execution against the `RECENT_ROOT_ADDRESS` predeploy.

## Implemented (this branch)

| Piece | Detail |
|-------|--------|
| Opcode `RECENTROOTREFLOAD` | `0xB5` (see divergence). Stack `[field, index]` (field on top); `field` 0→source_id, 1→slot, 2→root; gas 3; halts if `index ≥ len` or `field > 2`; reads only the envelope. |
| Envelope field | `recent_root_references: Vec<RecentRootReference>` — last RLP field (the **10th** ethrex field, after `blob_versioned_hashes`; the spec calls it the 9th because it omits ethrex's `signatures`). `RecentRootReference { source_id: H256, slot: u64, root: H256 }`, RLP `[source_id, slot, root]`, rkyv-derived. Covered by `compute_sig_hash` (not elided in VERIFY frames). |
| Static validation | `len ≤ FRAME_TX_MAX_RECENT_ROOT_REFERENCES (16)`; 3-tuple shape + 32-byte `source_id`/`root` + `slot < 2**64` enforced by the type/RLP. |
| TXPARAM `0x0F` | `len(recent_root_references)` (see spec-bug note). |
| `RECENT_ROOT_ADDRESS` | predeploy at `0x…8272` (`crates/vm/system_contracts.rs`). |
| Gas | `RECENTROOTREFLOAD = 3`; `RECENT_ROOT_REFERENCE_ADDRESS_GAS = ACCESS_LIST_ADDRESS_COST`; `RECENT_ROOT_REFERENCE_GAS = ACCESS_LIST_STORAGE_KEY_COST + 2*30 + 7*6`. Both the intrinsic reference gas **and** the EIP-7623 calldata cost over `rlp(recent_root_references)` are added to `FrameTransaction::total_gas_limit` (the references travel in the frame-tx envelope, so their RLP bytes are charged at the 4/16-per-byte calldata rate like `frames`/`signatures`). |

## Divergences from the draft spec

1. **`RECENTROOTREFLOAD` at `0xB5`, not `0xB4`.** The spec's Constants table assigns `0xB4`, which collides with EIP-8141's shipped `SIGPARAM = 0xB4`. ethrex uses the next free byte `0xB5`. Flag upstream.
2. **TXPARAM index `0x0F`** (not `0x0D`). The spec is internally inconsistent — its Constants table says `0x0F`, its summary table says `0x0D`. ethrex uses `0x0F` (Constants table is authoritative). An upstream fix is warranted.
3. **`RECENT_ROOT_ADDRESS = 0x…8272`** — the spec leaves it `TBD`; ethrex uses the `0x…<EIP>` convention.
4. **`RECENT_ROOT_CODE` handled natively** — the spec leaves the bytecode `TBD`; rather than invent bytecode, ethrex handles the 64-byte write (`salt ‖ root`, value 0, static-revert, direct-call-only) as a VM-level special case (see below).
5. **`source_id = keccak256(pad32(caller) ‖ salt)`** — the draft never defines how the write side derives `source_id` (it falls inside the TBD `RECENT_ROOT_CODE`). ethrex derives it from the caller and the salt: caller-authenticated (nobody can write into another caller's namespace) and salt-differentiated (a caller gets as many namespaces as it needs — this is what the 64-byte `salt ‖ root` calldata format exists for). A referencing transaction declares the same `source_id` in its envelope. **Needs upstream ratification** — a client deriving `source_id` differently would fork on any reference-carrying tx.
6. **Native write gas = flat `RECENT_ROOT_WRITE_GAS` (22100)** — the spec's gas schedule is TBD with the bytecode. Priced as a cold created-slot SSTORE (20000 + 2100); flat pricing slightly over-charges ring-entry overwrites (the safe direction). Failed calls revert with the forwarded gas returned.
7. **`DELEGATECALL`/`CALLCODE`/7702-delegation to the predeploy revert** — the spec says such calls "MUST NOT write"; ethrex picks revert over no-op-success (conservative, and closest to bytecode running against a foreign storage context).
8. **Write journaling follows plain storage semantics** — a write inside a reverting frame or atomic batch rolls back with it, exactly as a real bytecode `SSTORE` would. (Contrast EIP-8250 keyed-nonce consumption, whose spec explicitly mandates revert-survival.)
9. **Top-level EOA transactions sent directly `to = 0x…8272` do not write** — the native handler intercepts frame targets and CALL-family invocations (any contract can write, from any tx type), but not the initial call frame of a plain EOA transaction, which lands on the empty-code account as a no-op success. Use a frame or a contract call to write.

## Consensus core (implemented)

The full consensus path is wired; the interim reject-all gate is gone.

- **Reference validity** (`execute_frame_tx`, before the frame loop): each reference must satisfy `1 ≤ current_slot − slot ≤ 8191` (`current_slot` from EIP-7843 `env.slot_number`, checked in U256, never `block.timestamp`) and `RECENT_ROOT_ADDRESS[storage_key] == entry_hash`. An invalid reference invalidates the transaction — and on import the block — before any frame executes. `entry_hash` commits to the **raw** slot, so an entry overwritten by an aliasing newer slot (same `slot mod 8192`) can never satisfy a stale reference; this closes the forged-roots soundness hole, and `RECENTROOTREFLOAD` now serves only verified roots.
- **Hash derivation** (`RecentRootReference::{entry_hash, storage_key}` in ethrex-common): `entry_hash = keccak256(keccak256("RECENT_ROOT_ENTRY") ‖ source_id ‖ uint64_be(slot) ‖ root)`, `storage_key = keccak256(keccak256("RECENT_ROOT_STORAGE") ‖ source_id ‖ uint64_be(slot mod 8192))`. One definition shared by the validity check, the native write, and the mempool policy — a natively written root always validates its own reference.
- **Access warming**: the predeploy address and every reference's storage key are warmed in the validity pass (the per-reference intrinsic gas prepaid this).
- **Predeploy install** (`install_recent_root_code`): nonce 1, empty code, balance preserved; installed at the Hegota gate from both `prepare_block` and `apply_system_calls`, so the account exists before any tx executes in the first Hegota block.
- **Native write**: a frame targeting the predeploy (empty code → default-code path) or any CALL-family invocation (intercepted in `generic_call`) with exactly 64 bytes `salt ‖ root` and zero value commits the current slot's entry under `source_id = keccak256(pad32(caller) ‖ salt)`. Static contexts, delegate-style calls, and malformed calls revert. Last-write-wins follows from storage-write ordering.
- **Mempool/builder policy** (`validate_transaction`): admission-time rejection of too-new (`slot ≥ head_slot + 1`), expired (window exceeded), and uncommitted-root references, asserted explicitly against head state (the mempool's validation-prefix simulation never reaches `execute_frame_tx`, so the storage assertion is duplicated there by design). References are re-validated only at admission; head-change eviction is deferred like the other 8250 mempool niceties.

## ⚠ Remaining / flagged for devnet validation

- **Cross-client interop** on the Hegotá devnet: the `source_id` derivation (divergence #5), the write gas convention (#6), and the hash byte layouts need a second client / spec ratification before any public network relies on them. Any mismatch is a hard fork on the first reference-carrying transaction.
- **Head-change recheck/eviction** of admitted reference-carrying txs (admission-only today).
- Upstream spec gaps to flag: `RECENT_ROOT_CODE` bytecode + gas, `source_id` derivation, opcode byte (`0xB4` collision), TXPARAM index inconsistency.
