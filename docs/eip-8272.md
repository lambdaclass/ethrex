Implementation notes for EIP-8272 in ethrex. Builds on EIP-8141 frame transactions and EIP-7843 `SLOTNUM` (both already present); activates under `Fork::Hegota`.

## Overview

EIP-8272 lets a frame transaction declare verified **recent roots** â€” `(source_id, slot, root)` tuples â€” in its signed envelope, so validation code (e.g. privacy proofs) can reference a recent commitment without reading another account's mutable storage. References are checked for validity at block execution against the `RECENT_ROOT_ADDRESS` predeploy.

## Divergences from the draft spec

1. **`RECENTROOTREFLOAD` at `0xB5`, not `0xB4`.** The spec's Constants table assigns `0xB4`, which collides with EIP-8141's shipped `SIGPARAM = 0xB4`. ethrex uses the next free byte `0xB5`. Flag upstream.
2. **TXPARAM index `0x0F`** (not `0x0D`). The spec is internally inconsistent â€” its Constants table says `0x0F`, its summary table says `0x0D`. ethrex uses `0x0F` (Constants table is authoritative). An upstream fix is warranted.
3. **`RECENT_ROOT_ADDRESS = 0xâ€¦8272`** â€” the spec leaves it `TBD`; ethrex uses the `0xâ€¦<EIP>` convention.
4. **`RECENT_ROOT_CODE` handled natively** â€” the spec leaves the bytecode `TBD`; rather than invent bytecode, ethrex handles the 64-byte write (`salt â€– root`, value 0, static-revert, direct-call-only) as a VM-level special case (see below).
