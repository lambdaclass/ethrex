# EIP-7906: Transaction Assertions (POST_TX frames)

> **Spec target:** ethereum/EIPs `EIPS/eip-7906.md` (Draft, Standards Track: Core; requires EIP-8141). Read against `eips.ethereum.org` master on 2026-06-24, **including the two merged updates** PR #11829 (POST_TX frame mode) and PR #11830 (TXDIFF opcode). Pin the exact merge commit per the change task once frozen.

Implementation notes for EIP-7906 in ethrex. Builds on EIP-8141 frame transactions; activates under `Fork::Hegota`.

## Overview

EIP-7906 lets a frame transaction append **assertion frames** that inspect the whole transaction's effects after the body has executed and **revert the entire transaction** if an invariant is violated (e.g. "this tx made no hidden ERC-20 approval"). It adds:

- A new frame mode **POST_TX** (`mode = 3`): a trailing-suffix frame, run read-only (STATICCALL, caller = ENTRY_POINT) after the tx body, whose revert invalidates the whole transaction body.
- Three introspection opcodes â€” **TXTRACE**, **EVENTDATACOPY**, **TXDIFF** â€” that read the transaction-scoped state/event diff and gas context. They are valid **only inside a POST_TX frame's call subtree**; anywhere else they exceptional-halt.

## Frame-mode namespace (Hegota)

| Mode | Value | EIP |

## Implemented (this branch)

| Piece | Detail |
|-------|--------|
| `FrameMode::PostTx = 3` | `from_u8(3)`, `From<FrameMode> for u8`, dispatch in the frame loop â†’ `(caller = entry_point, static = true)`; rejected in the validation-prefix simulation (POST_TX is body, never prefix). |
| Trailing-suffix rule | Static check: once any frame is POST_TX, every later frame must be POST_TX (`"POST_TX frames must form a contiguous trailing suffix"`). |
| Whole-body revert | A reverted POST_TX frame sets `tx_invalid` (both the atomic-batch-revert and non-batch-revert sites in `vm.rs`), overriding atomic-batch unrolling â€” same tx-level invalidation as a reverted VERIFY frame. |
| Opcode gating | `require_post_tx_frame(vm)` gates TXTRACE / EVENTDATACOPY / TXDIFF: halts (`InvalidOpcode`) unless the enclosing tx frame's mode is POST_TX. Holds for nested calls in the POST_TX subtree (`current_frame_index` tracks the enclosing tx frame). |
| `TXTRACE` | `0xB5`. Stack `[in2, param]` (in2 on top). Reads the tx-scoped diff: balance/slot/code-deployment changes (counts + indexed fields), ordered event fields, gas pre-charge (`0x14`) and gas payer (`0x15`). Gas 100. |
| `EVENTDATACOPY` | `0xB6`. Stack `[event_index, memOff, dataOff, length]`. Copies an emitted event's data into memory (past-the-end reads halt; no zero-fill). Gas mirrors CALLDATACOPY. |
| `TXDIFF` | `0xB7`. Keyed lookup; stack `[param, address, in3]` (param on top, then address, then `in3`). `param` 0x00 slot_before / 0x01 slot_after / 0x02 balance_before / 0x03 balance_after / 0x04 codehash_before / 0x05 codehash_after. `in3` is the slot key for the slot params and must be zero for the scalar (balance/codehash) params. "before" reads the tx prestate (`initial_accounts_state`), "after" the live post-body state (`current_accounts_state`); a key the tx never modified reads the live value both ways; an undeployed account's codehash_before is the empty-Keccak hash. Reads load the account/slot into the diff caches but never trigger EIP-2929 warm/cold accounting. Gas 2100 (provisional). |

## Divergences from the draft spec

1. **TXDIFF gas `2100` is provisional.** PR #11830 marks the gas cost TBD; ethrex prices a keyed before/after lookup as a cold SLOAD (`SLOAD_COLD_DYNAMIC = 2100`) rather than the flat warm `TXTRACE` cost, since the lookup may touch a cold account/slot. **TXTRACE gas `100` is likewise the EIP's own provisional example value.**
2. **TXDIFF "after" reads the live state via the diff caches**, not a separately materialized post-tx snapshot. Inside a POST_TX frame the live `current_accounts_state` already reflects the executed body (POST_TX is the trailing suffix), so before (`initial`) vs after (`current`) is the whole-tx diff. Equivalent to the spec intent; flagged for cross-client confirmation of edge cases.

## âš  Remaining / flagged for devnet validation

- **Whole-body-revert payment semantics.** The spec states a failed POST_TX assertion "unconditionally invalidates the entire transaction, including any gas payment already approved", while also describing the validation prefix as not reverted in a mempool-compatible way. ethrex's consensus path reverts the whole tx via the tx-level backup (the tx is excluded), setting `tx_invalid`. The precise *included-but-reverted vs excluded* receipt semantics and the prefix-payment interaction are **flagged for devnet + cross-client interop confirmation** (see the `NOTE` in `vm.rs` at the non-batch-revert site).
- **POST_TX gas budgeting.** In the frame-tx execution path, a body-frame SSTORE was observed to consume substantially more than the bare opcode cost in the integration harness; POST_TX assertion frames should be budgeted generously until frame-tx gas accounting is confirmed on the devnet. Tracked separately from EIP-7906 (frame-tx gas accounting, not assertion logic).
- **TXDIFF gas** finalization once PR #11830 freezes the value.
- Cross-client interop on the POST_TX trailing-suffix structural rule and the gating of the three opcodes outside POST_TX.
