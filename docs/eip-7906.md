# EIP-7906: Transaction Assertions (POST_TX frames)

> **Spec target:** ethereum/EIPs `EIPS/eip-7906.md` (Draft, Standards Track: Core; requires EIP-8141). Read against `eips.ethereum.org` master on 2026-06-24, **including the two merged updates** PR #11829 (POST_TX frame mode) and PR #11830 (TXDIFF opcode). Pin the exact merge commit once it is frozen.

Implementation notes for EIP-7906 in ethrex. Builds on EIP-8141 frame transactions; activates under `Fork::Hegota`.

## Overview

EIP-7906 lets a frame transaction append **assertion frames** that inspect the whole transaction's effects after the body has executed and **revert the entire transaction** if an invariant is violated (e.g. "this tx made no hidden ERC-20 approval"). It adds:

- A new frame mode **POST_TX** (`mode = 3`): a trailing-suffix frame, run read-only (STATICCALL, caller = ENTRY_POINT) after the tx body, whose revert invalidates the whole transaction body.
- Three introspection opcodes — **TXTRACE**, **EVENTDATACOPY**, **TXDIFF** — that read the transaction-scoped state/event diff and gas context. They are valid **only inside a POST_TX frame's call subtree**; anywhere else they exceptional-halt.

## Frame-mode namespace (Hegota)

| Mode | Value | EIP |
|------|-------|-----|
| DEFAULT | 0 | 8141 |
| VERIFY | 1 | 8141 |
| SENDER | 2 | 8141 |
| **POST_TX** | **3** | **7906** |
| DEP_VERIFY | 4 | 8288 (deferred) |

`validate_static_constraints` admits modes `0..=3` (`frame.mode >= 4` is rejected). When EIP-8288 resumes, relax to `>= 5`.

## Implemented (this branch)

| Piece | Detail |
|-------|--------|
| `FrameMode::PostTx = 3` | `from_u8(3)`, `From<FrameMode> for u8`, dispatch in the frame loop → `(caller = entry_point, static = true)`; rejected in the validation-prefix simulation (POST_TX is body, never prefix). |
| Trailing-suffix rule | Static check: once any frame is POST_TX, every later frame must be POST_TX (`"POST_TX frames must form a contiguous trailing suffix"`). |
| Whole-body revert | A reverted POST_TX frame sets `tx_invalid` (both the atomic-batch-revert and non-batch-revert sites in `vm.rs`), overriding atomic-batch unrolling — same tx-level invalidation as a reverted VERIFY frame. |
| Opcode gating | `require_post_tx_frame(vm)` gates TXTRACE / EVENTDATACOPY / TXDIFF: halts (`InvalidOpcode`) unless the enclosing tx frame's mode is POST_TX. Holds for nested calls in the POST_TX subtree (`current_frame_index` tracks the enclosing tx frame). |
| `TXTRACE` | `0xB5`. Stack `[in2, param]` (in2 on top). Reads the tx-scoped diff: balance/slot/code-deployment changes (counts + indexed fields), ordered event fields, gas pre-charge (`0x14`) and gas payer (`0x15`). Gas 100. |
| `EVENTDATACOPY` | `0xB6`. Stack `[event_index, memOff, dataOff, length]`. Copies an emitted event's data into memory (past-the-end reads halt; no zero-fill). Gas mirrors CALLDATACOPY. |
| `TXDIFF` | `0xB7`. Keyed lookup; stack `[param, address, in3]` (param on top, then address, then `in3`). `param` 0x00 slot_before / 0x01 slot_after / 0x02 balance_before / 0x03 balance_after / 0x04 codehash_before / 0x05 codehash_after. `in3` is the slot key for the slot params and must be zero for the scalar (balance/codehash) params. "before" reads the tx prestate (`initial_accounts_state`), "after" the live post-body state (`current_accounts_state`); a key the tx never modified reads the live value both ways; an undeployed account's codehash_before is the empty-Keccak hash. Reads load the account/slot into the diff caches but never trigger EIP-2929 warm/cold accounting. Gas 2100 (provisional). |

## Divergences from the draft spec

1. **TXDIFF gas `2100` is provisional.** PR #11830 marks the gas cost TBD; ethrex prices a keyed before/after lookup as a cold SLOAD (`SLOAD_COLD_DYNAMIC = 2100`) rather than the flat warm `TXTRACE` cost, since the lookup may touch a cold account/slot. **TXTRACE gas `100` is likewise the EIP's own provisional example value.**
2. **TXDIFF "after" reads the live state via the diff caches**, not a separately materialized post-tx snapshot. Inside a POST_TX frame the live `current_accounts_state` already reflects the executed body (POST_TX is the trailing suffix), so before (`initial`) vs after (`current`) is the whole-tx diff. Equivalent to the spec intent; flagged for cross-client confirmation of edge cases.

## ⚠ Remaining / flagged for devnet validation

- **Whole-body-revert: behavior is internally consistent; only the receipt representation is underspecified.** A reverted POST_TX frame sets `tx_invalid` and the tx is **excluded** from the block via the tx-level backup rollback (every committed frame's state *and* any APPROVE-granted gas payment are undone; `execute()` returns `InvalidFrameTransaction`, the payload builder drops the tx). This is the **same code path as a reverted VERIFY frame** (shared `matches!(mode, FrameMode::Verify | FrameMode::PostTx)` guard) and is consistent with EIP-7906's "unconditionally invalidates the entire transaction, including any gas payment already approved" — "invalidate" = exclude, with the approved payment rolled back. What genuinely needs cross-client/devnet confirmation: (1) **receipt representation** — exclude-entirely (ethrex's choice: no receipt, not in the block body) vs include-but-mark-reverted (a status-0 receipt occupying a block slot), which affects receipts-root/block-hash agreement; (2) the **validation-prefix payment interaction** — the spec also calls the prefix "not reverted in a mempool-compatible way", in tension with rolling the approved payment back (ethrex rolls it back, so the payer pays nothing); (3) the **anti-DoS** angle (a builder bears the execution cost of POST_TX-reverting txs with no compensation if the payer pays nothing). See the `NOTE` in `vm.rs` at the non-batch-revert site.
- **CONFIRMED BUG (EIP-8141 frame-tx gas accounting, not EIP-7906) — needs a dedicated fix.** A storage write inside any frame is over-billed by the full EIP-8037 *state-gas* amount (~97,920 per `SSTORE_SET`). Root cause: `execute()` dispatches frame txs to `execute_frame_tx()` **before** `prepare_execution()`, so `add_intrinsic_gas()` never funds the per-tx `state_gas_reservoir` (stays 0); when a frame's SSTORE calls `increase_state_gas()`, the entire state portion **spills into `gas_remaining`** instead of being drawn from the reservoir, and the frame report hardcodes `state_gas_used: 0`. A normal tx draws the same state gas from the reservoir (2D split), so an identical SSTORE costs ~98k less outside a frame — a real divergence, and two cold SSTOREs already exhaust a 200k-gas frame. **Recommended fix (on `eip-8141-1`, not here):** fund a per-frame `state_gas_reservoir` from the frame's `gas_limit` headroom before `run_execution()` and reconcile it afterward (mirroring `add_intrinsic_gas` + `finalize_execution`), folding state gas into the report's `state_gas_used`. **First confirm with the spec/EELS whether frame txs participate in the EIP-8037 2D state-gas split at all** — the correct fix shape depends on it. Evidence: `vm.rs` (frame dispatch before prepare; reservoir init 0; `increase_state_gas` spill; `state_gas_used: 0`), `utils.rs` (`add_intrinsic_gas` — the only funder, never reached on the frame path). Until fixed, budget body/POST_TX frames generously.
- **TXDIFF gas** finalization once PR #11830 freezes the value.
- Cross-client interop on the POST_TX trailing-suffix structural rule and the gating of the three opcodes outside POST_TX.
